# Deployment

Back2u ships as **one backend on Render** (free tier) and **four static frontends on Vercel**.
MongoDB is hosted separately on **MongoDB Atlas** (free M0). Everything below can run at $0 —
paid tiers are only needed to remove the free-tier limitations noted at the end.

```
MongoDB Atlas (M0, free)  ──▶  Render web service: back2u-api (free, Docker)
                                      ▲
Vercel (4 static SPAs) ────────────── │  (calls VITE_API_URL, must be in CORS_ORIGINS)
  • website      (marketing)
  • client-web   (the app)
  • admin        (operations console)
  • partner      (partner portal)
```

---

## 1. MongoDB Atlas (free)

1. Create a free **M0** cluster at https://cloud.mongodb.com.
2. Database Access → add a user (username + password).
3. Network Access → allow `0.0.0.0/0` (Render free has no static egress IP).
4. Copy the `mongodb+srv://…` connection string — this is `MONGO_URI`.

## 2. Render — API (`render.yaml` blueprint)

The blueprint provisions a single **free** Docker web service. The API runs job
handlers inline and an in-process scheduler, so **no separate worker and no Redis
are needed** on the free tier.

1. Render → **New → Blueprint**, point it at this repo. It reads `render.yaml`.
2. Fill the `sync: false` env vars in the dashboard. Only these are required to boot:
   - `MONGO_URI` — from Atlas above
   - `API_PUBLIC_URL` — this service's own URL, e.g. `https://back2u-api.onrender.com`
   - `APP_PUBLIC_URL` — the client-web Vercel URL (used in emails / share links)
   - `CORS_ORIGINS` — comma-separated Vercel origins, e.g.
     `https://back2u.vercel.app,https://back2u-admin.vercel.app,https://back2u-partner.vercel.app,https://back2u-site.vercel.app`
   - `CORS_PREVIEW_REGEX` _(optional)_ — `^https://.*\.vercel\.app$` to allow preview URLs
   - `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` are auto-generated by Render.
3. Everything else (`ANTHROPIC_API_KEY`, `MAPBOX_TOKEN`, `CLOUDINARY_*`, `RESEND_*`,
   `VAPID_*`, `TWILIO_*`, `HUBTEL_*`, `PAYSTACK_*`, `EXPO_ACCESS_TOKEN`, `VAULT_MASTER_KEY`,
   `SENTRY_DSN`) is **optional** — leave blank to disable that feature. `MAPBOX_TOKEN`
   powers `/v1/geo/search`, which the partner-signup location picker uses.

Health check: `GET /health`. Migrations run automatically on each deploy
(`node dist/cli/migrate.js && node dist/server.js`).

## 3. Vercel — the four frontends

Each app has its own `vercel.json` (SPA rewrites + monorepo build). Create **one Vercel
project per app**, all pointing at this same repo:

| Vercel project | Root Directory    | Env vars to set                                                                                                                         |
| -------------- | ----------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| website        | `apps/website`    | `VITE_API_URL`, `VITE_APP_URL` (client-web URL), `VITE_MAPBOX_TOKEN`, `VITE_APP_STORE_URL`, `VITE_PLAY_STORE_URL`, `VITE_CONTACT_EMAIL` |
| client-web     | `apps/client-web` | `VITE_API_URL`, `VITE_APP_URL` (its own URL), `VITE_MAPBOX_TOKEN`                                                                       |
| admin          | `apps/admin`      | `VITE_API_URL`                                                                                                                          |
| partner        | `apps/partner`    | `VITE_API_URL`, `VITE_APP_PUBLIC_URL`                                                                                                   |

- Set **Root Directory** to the app folder in each project's settings. Vercel detects the
  pnpm workspace and the bundled `vercel.json` runs `pnpm turbo run build --filter=@back2u/<app>`
  from the repo root (so shared packages build first) and serves `dist/` with an SPA fallback.
- `VITE_API_URL` = the Render API URL (e.g. `https://back2u-api.onrender.com`). It must also
  appear in the API's `CORS_ORIGINS` (see step 2).
- Vite env vars are build-time — redeploy the frontend after changing them.

## 4. Deploy order

1. Atlas cluster → get `MONGO_URI`.
2. Render blueprint → set `MONGO_URI` + placeholder URLs → first deploy → note the API URL.
3. Vercel: create the 4 projects → set `VITE_API_URL` to the Render URL → deploy → note their URLs.
4. Back in Render: set `CORS_ORIGINS`, `API_PUBLIC_URL`, `APP_PUBLIC_URL` to the real URLs → redeploy.

## Free-tier caveats & upgrade path

- **Render free web services sleep** after ~15 min idle and cold-start (~30–60s) on the next
  request. While asleep, the in-process scheduler (expiry reminders, auction close) does not run.
- To make the backend always-on and the queue durable: upgrade the web service to a paid plan,
  add a Render **Key Value (Redis)** instance and set `REDIS_URL`, and (optionally) add a second
  `type: worker` service running `node dist/worker.js` — the code already supports BullMQ when
  `REDIS_URL` is present.
- The mobile app (`apps/client-mobile`, Expo) is not deployed here — build it with EAS separately
  and point `EXPO_PUBLIC_API_URL` at the Render API.
